## Why Software Testers Use This Tool

Writing Playwright tests manually is slow and repetitive. This tool eliminates that entirely.

**You write a spec MD file (plain English + numbered steps), and the tool generates hundreds of real browser tests automatically.**

| Without this tool | With this tool |
|---|---|
| Write thousands of lines of Playwright code by hand | Maintain one Markdown file per page |
| A week of work per feature | Tests appear in minutes |
| Tests drift out of sync with the spec | Spec IS the source of truth — tests regenerate automatically |
| Security, accessibility, load time checks skipped | Every test type generated by default, every time |
| Only works for one codebase | Drop any project's specs in — swap env vars |
| Tests simulate fake data | Tests use real accounts, real OTP, real database entries |

**Every spec MD file automatically produces:**
- UI element visibility checks
- User flow walkthroughs (from numbered step lists)
- Requirement regression tests
- Edge case coverage (from validation bullet points)
- XSS and SQL injection security probes
- Load time assertions
- Boundary value and invalid input tests
- **Cross-account data verification** (teacher data visible to student, and vice versa)

## Quick Start (5 minutes)

### 1. Clone

```bash
git clone https://github.com/Qatarat/mehad-automation
cd mehad-automation
```

### 2. Install dependencies

```bash
pip install -r requirements.txt
playwright install chromium
```

### 3. Set credentials — use real temp phone numbers

```bash
export BASE_URL="https://dev.mehadedu.com/en"

# Teacher test account (real phone number registered in the system):
export TEACHER_PHONE="98976564"
export TEACHER_OTP="123456"
export TEACHER_COUNTRY="+880"

# Student test account (different real phone number):
export STUDENT_PHONE="98765432"
export STUDENT_OTP="123456"
export STUDENT_COUNTRY="+880"

# Shared OTP — dev environment uses 123456 as bypass for test numbers
export TEST_OTP="123456"
export TEST_COUNTRY="+880"
```

> **Why real phone numbers?**  
> The tests verify data flows into the real database. If the teacher creates a session, the student account must see it. If the student enrolls, the teacher account must show the enrollment. This only works with real registered accounts — not simulated data.

### 4. Run all tests

```bash
pytest tests/test_specs_all.py -v
```

Tests auto-regenerate from specs every time you run — if you changed any `.md` file since the last run, tests rebuild before pytest collects.

## Full Teacher–Student E2E Flow

The most important test path in the platform. The Fagun platform verifies the complete business cycle end-to-end:

```
Teacher Account                    Student Account
─────────────────                  ─────────────────
1. Sign up with real phone         1. Sign up with different phone
2. Complete tutor profile          2. Browse Find Tutors page
3. Create Group Session            3. Find the teacher's session
   (3-step: date/time/course)      4. Click Enroll Now
4. Set 1-to-1 availability         5. Complete payment (test card)
   (calendar slots)                6. Booking appears in My Bookings
5. See enrolled students list      7. Join classroom at session time
6. Join classroom                  8. After session: view recording
7. Earnings → Pending Balance      9. Leave review/rating
```

**What gets verified at each step:**
- After teacher creates session → session appears on `/en/find-tutors` for the student
- After student enrolls → teacher's dashboard shows the enrolled student
- After payment → booking record in student's account matches teacher's booking record
- After session → recording available to both parties
- Earnings: pending balance increases after completed paid session

## How It Works

```
specs/*.md  ←  you write this (plain English, numbered steps)
     ↓
ai_engine/spec_parser.py   ←  extracts URL, flows, edge cases, requirements
ai_engine/spec_compiler.py ←  extracts selectors, test data
     ↓
scripts/validate_all_specs.py  ←  generates tests/test_specs_all.py
     ↓
tests/test_specs_all.py        ←  700+ tests across 41 specs
     ↓
pytest + Playwright            ←  runs against dev.mehadedu.com/en
     ↓
reports/                       ←  HTML report, JSON results, screenshots
```

The `conftest.py` hook watches for spec changes. If any `.md` file is newer than the generated test file, tests auto-regenerate before pytest collects. **You never run the generator manually.**

## Auto-Discovery: Drop a File, Get Tests

Any `.md` file you add to `specs/` is picked up on the next `pytest` run. No code changes. No registration.

```bash
# Add a new page spec
cp specs/TEMPLATE.md specs/tutor_dashboard.md
# Edit it — describe the page with numbered steps
pytest tests/test_specs_all.py -k "TutorDashboard" -v
# Tests are auto-generated and run immediately
```

Files named `TEMPLATE.md`, `README.md`, or `EXAMPLE.md` are skipped.

## Spec Format — Two Styles Supported

### Style 1: Structured (full coverage, recommended for new specs)

```markdown
# Login Page

**URL:** `https://dev.mehadedu.com/en`

## UI Elements
| Element          | Selector                           | Notes    |
|------------------|------------------------------------|----------|
| Login button     | `button:has-text("Log In")`        | Required |
| Phone input      | `input[type="tel"]`                | Required |
| Send Code button | `button:has-text("Send Code")`     | Required |

## User Flows
### Flow 1: Happy path login
1. Navigate to homepage
2. Click Login button
3. Select country code +880
4. Enter phone number from TEACHER_PHONE env var
5. Click Send Code
6. Enter OTP from TEST_OTP env var
7. Verify redirect to dashboard

## Edge Cases
| EC-01 | Empty phone number | Error shown |
| EC-02 | Wrong OTP code     | Error shown |

## Requirements
- REQ-01: Login button visible in header
- REQ-02: OTP must be 6 digits
```

### Style 2: Prose/Steps (Mehad's current format — fully supported)

```markdown
# Tutor Signup Process

## Steps to Reproduce Tutor Signup

1. Go to https://dev.mehadedu.com/en
2. Click "Become a Tutor" button from header
3. Click "Apply Now" button
4. Enter country code and phone number
5. Click "Send Code" and enter the OTP

### Validation
- First Name: maximum 30 characters
- Email: must be unique and valid format
- Bio: maximum 500 characters
```

Both formats produce full test coverage.

## Authentication — WhatsApp OTP with Real Accounts

Mehad uses WhatsApp OTP auth (phone number + 6-digit code). The Fagun platform maintains **separate sessions for teacher and student** — both log in at the start of the test suite and all tests run in the correct account context.

```bash
# Teacher account (tutor — registers under /en/become-tutor):
TEACHER_PHONE="98976564"      # real phone registered as tutor
TEACHER_OTP="123456"          # dev OTP bypass for test accounts
TEACHER_COUNTRY="+880"        # Bangladesh (+880)

# Student account (registers under /en):
STUDENT_PHONE="98765432"      # different real phone registered as student
STUDENT_OTP="123456"
STUDENT_COUNTRY="+880"

# Override login page if needed:
LOGIN_URL="https://dev.mehadedu.com/en"
```

**Important:** After signup, always use the **same phone number to log in**. Do not mix temp/demo accounts. The test suite verifies that:
- Data created in the teacher account is visible from the student account
- Data created in the student account (enrollment, booking) is visible in the teacher account
- Both accounts reflect the same real database state

## Data Integrity Verification

Every test that creates or modifies data includes a verification step — the data must appear in the UI, which confirms it was written to the database.

| Action | Verified in |
|---|---|
| Teacher creates group session | Session appears on student's `/en/find-tutors` page |
| Student enrolls in session | Enrollment shows in teacher's students list |
| Student books 1-to-1 slot | Booking appears in both teacher and student dashboards |
| Payment completed | Transaction visible in student's payment history and teacher's earnings |
| Session completed | Recording link available to both parties |
| Teacher updates profile | Updated info shown on student-facing tutor card |

Tests are **not simulated** — they interact with the real application and assert the same data appears in both account views.

## Using This Tool for Any Project

Replace the Mehad specs with your own:

```bash
# Remove the existing specs
rm specs/*.md

# Add your own
cp specs/TEMPLATE.md specs/dashboard.md
cp specs/TEMPLATE.md specs/checkout.md
# Edit each file — describe the page with numbered steps

# Point at your app
export BASE_URL="https://staging.your-app.com"
export LOGIN_URL="https://staging.your-app.com/login"
export TEST_EMAIL="qa@yourcompany.com"
export TEST_PASSWORD="YourQAPassword"

# Run
pytest tests/test_specs_all.py -v
```

The tool name "Fagun Autonomous QA Platform" is fixed — it does not change per project. Only the `BASE_URL` and credentials change.

The platform generates tests **without any AI API key**. AI is used for enhanced test generation and self-healing:

## Troubleshooting

**Auth fails — `RuntimeError: Auth setup failed`**
- Set `TEACHER_PHONE` + `TEST_OTP` + `TEST_COUNTRY` and verify the account is registered in dev
- Set `HEADED=1` to watch the auth flow in the browser
- The phone number must be a real registered account (not a random number)

**`playwright: Executable doesn't exist`**
```bash
playwright install chromium
```

**Load time tests failing**
```bash
LOAD_TIME_LIMIT=30 pytest tests/test_specs_all.py -v
```

**Teacher data not visible to student**
- Confirm both `TEACHER_PHONE` and `STUDENT_PHONE` are different real accounts
- The teacher account must have completed the tutor application (status: approved)

**New spec not generating tests**
```bash
touch specs/your_spec.md && pytest tests/test_specs_all.py -v
```
